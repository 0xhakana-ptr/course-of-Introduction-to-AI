# 2026-05-11 Backend Agent Capability Improvement Plan

副标题：后端 Agent 能力完善方案 / Backend Agent Capability Improvement Plan

> 2026-05-12 架构修订说明：
> 本文件中的 P0、P1 已完成且继续有效；从 P2 开始，后续方向已被
> `docs/plans/2026-05/20260512-1.md` 接管。新的原则是：现有顶层 Agent Loop

## 1. 背景

截至 `20260511-4`，后端已经完成以下基础收口：

- `/chat` 默认主路径已经收敛到单 Agent Loop。
- 旧 route graph、`AGENT_RUNTIME_MODE` 和 legacy diagnostics 已移除。
- 文件覆盖确认、图外异常终态事件、历史 mock 隔离、旧节点 metadata 清理已完成。
- `backend/tests/` 已按模块边界分类，完整后端测试保持通过。

当前后端已经具备“稳定进入 Agent Loop 并执行基础 action”的能力，但距离真正的桌宠 Agent 还差一层关键能力：

- 现在仍偏“规则规划 + 单次 action + 少量队列”。
- 工具循环还不够像真正 Agent，面对复杂任务时容易卡住、提前结束或只创建 run。
- 前端虽然能收到节点事件，但还需要更稳定的“节点开始 JSON / 动作 JSON / 终态 JSON”契约。
- token 成本、图片/URL 输入、上下文裁剪和工具选择边界还需要继续强化。

## 2. 总目标

把当前后端从“稳定 Agent Loop 骨架”推进到“可实际完成桌宠任务的工具型 Agent”。

核心原则：

- `LongCat / MiniMax / OpenAI-compatible LLM` 只是大模型脑子，不是 Agent 本体。
- Agent 本体是 `LangGraph Agent Loop + Action Registry + Tools + State + Messaging`。
- 不再回到旧 route graph，也不新增横向固定路由分支。
- 所有新能力优先落到 `agent_workflow/loop/`、`agent_workflow/actions/`、`tools/`、`services/` 和 `messaging/`。
- 优先做能被测试验证、能被前端稳定消费、能减少真实卡住问题的改动。

非目标：

- 不做多 Agent 协作编排。
- 不一次性重写整个后端。
- 不让 LLM 每轮都承担重规划，避免 token 成本失控。
- 不绕过 workspace 安全边界直接操作任意系统路径。

## 3. 当前主要问题

### 3.1 Agent Loop 还不够“Agent”

现状：

- `plan_node` 多数情况下仍由规则规划 action。
- `observe -> decide -> plan` 已存在，但复杂场景下还没有形成稳定的“根据观察继续选工具”能力。
- 写文件、读文件、列目录、run 创建、run 控制已经能跑，但“连续完成任务”的能力还偏弱。

影响：

- 用户让桌宠“创建文件并检查内容”“先看目录再写文件”“失败后换路径”等任务时，容易只做一步或退回 run 创建。

### 3.2 工具选择仍需更清晰的结构化契约

现状：

- workspace 工具规划已经有结构化模型，但 Agent Loop 内部还没有统一的 `ActionPlan` 契约。
- action queue、confirmation、workspace plan、run action plan 的字段来源不够统一。

影响：

- 后续接 LLM planner 时，容易再次变成散落字符串规则。
- 前端、diagnostics、测试难以稳定复盘“为什么选这个工具”。

### 3.3 前端事件还需要进一步标准化

现状：

- `workflow.node_entered`、`workflow.action_started/completed/failed`、`workflow.completed/failed` 已存在。
- 但队长提出的“不同 LangGraph 节点开头发一个 JSON”“动作控制也在节点开头发 JSON”还没有形成明确文档和稳定 schema。

影响：

- 前端能显示过程，但不一定能稳定区分“开始理解、开始规划、开始写文件、等待确认、执行失败、任务完成”。

### 3.4 token 成本和复杂输入处理还需要硬边界

现状：

- 已有上下文裁剪、会话摘要和基础 LLM fallback。
- 但图片、URL、长上下文、连续追问时的预算策略还不够明确。

影响：

- 容易出现“几个问题烧很多 token”。
- 图片或 URL 输入可能被当作普通文本，导致无效上下文进入模型。

### P1：统一 Agent Loop 内部 ActionPlan 契约

状态：

```text

```

- `backend/app/agent_workflow/runtime/`

- 将 `workspace_tool_plan`、`next_action_queue`、`confirmation_required` 等字段统一到 plan details。

验收标准：

- 现有 241 个测试继续通过。
- 新增测试覆盖 chat、workspace.write、workspace.read、run.create、ask_user_confirmation 的 plan 结构。

优先级：

```text
最高
```

### P2：强化 observe -> decide -> plan 的多步循环

状态：

```text
未开始
```

目标：

- 让 Agent Loop 在安全边界内真正支持简单多步任务。

优先支持场景：

- 创建文件后读取确认。
- 读取文件失败后给出明确下一步。
- 写文件遇到已存在时进入确认分支。
- 列目录后根据结果继续读取目标文件。
- run 创建后能给出明确下一步，而不是假装已经完成所有代码任务。

范围：

- `backend/app/agent_workflow/loop/agent_loop_graph.py`
- `backend/app/agent_workflow/runtime/models.py`
- `backend/app/agent_workflow/output/action_events.py`
- `backend/tests/agent_workflow/`
- `backend/tests/acceptance/`

任务：

- 让 `observation` 结构稳定包含 `action_name / ok / output / error / data / next_candidates`。
- 让 `decide_continue_node` 根据 observation 判断是否继续，而不是只看队列。
- 引入最小的 replan 策略：优先规则，不够时再考虑 LLM。
- 保留 `max_steps` 防死循环。

验收标准：

- 多步任务能在 2-3 步内完成。
- 失败任务不会卡住，能进入 `workflow.failed` 或明确的 `final.answer`。
- 不增加普通聊天 token 消耗。

优先级：

```text
高
```

### P3：接入轻量 LLM Planner，但不替代安全规则

目标：

- 当规则 planner 无法判断下一步，但任务明显属于工具型任务时，再调用 LLM 生成结构化 plan。

原则：

- LLM planner 只输出候选计划，不直接执行。
- 候选计划必须经过 Action Registry、路径安全、确认门、工具参数校验。
- 普通聊天不走 planner。
- 简单文件读写不必每次走 planner，优先规则快路径。

范围：

- `backend/app/agent_workflow/loop/`
- `backend/app/llm/client.py`
- `backend/app/tools/workspace_tool_models.py`
- `backend/tests/agent_workflow/`

任务：

- 设计 planner prompt，要求输出严格 JSON。
- 增加 JSON 解析失败 fallback。
- 将 planner 结果映射为 `ActionPlan`。
- 增加 token 预算限制和 planner 调用开关。

验收标准：

- 复杂工具请求能得到更合理计划。
- LLM 输出非法 JSON 时不崩溃。
- 未配置 LLM 时仍能使用规则 planner。

优先级：

```text
中高
```

## 5. 推荐执行顺序

```text
P0 -> P1 -> P2 -> P4 -> P3 -> P5 -> P6 -> P7
```

理由：

- `P0` 先冻结当前稳定基线。
- `P1` 先统一计划结构，否则后面多步循环和 LLM planner 会继续堆分支。
- `P2` 解决“Agent 是否真的能连续做事”的核心问题。
- `P4` 尽早对齐前端事件，避免后端做完后前端无法稳定展示。
- `P3` 放在 ActionPlan 和多步循环之后，避免 LLM planner 接进错误结构。
- `P5/P6/P7` 作为稳定性、成本和协作收尾。

## 7. 风险与控制

风险：

- 过早引入 LLM planner 会增加 token 成本，并可能让行为不稳定。
- 多步循环如果没有 `max_steps` 和明确 stop reason，可能重新造成前端卡住。
- 前端如果依赖 action completed 解除 loading，仍可能误判 workflow 未结束。

控制：

- 所有循环必须保留 `max_steps`。
- 所有高风险 action 必须走 confirmation gate。
- 所有 `/chat` turn 必须发 `workflow.completed` 或 `workflow.failed`。
- 普通聊天不进入 LLM planner。
- 每个阶段必须补测试后再进入下一阶段。
