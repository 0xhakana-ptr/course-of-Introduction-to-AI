# 2026-05-06 后端优化与完善方案

## 1. 当前后端状态判断

结合当前仓库最新后端代码，现阶段最重要的问题不是“功能太少”，而是“功能雏形已经不少，但运行边界、模块契约和文档状态还没有完全收口”。

当前主要风险如下：

1. 包导入策略不统一。
   目前 `backend/app/main.py`、`backend/app/services/chat_service.py`、`backend/app/services/run_service.py`、`backend/app/llm/client.py` 等模块同时存在脚本式导入和包式运行方式，导致 `uv run uvicorn backend.app.main:app --reload --port 8000` 这条官方启动链路不稳定。

2. 入口文件存在缺失依赖。
   `main.py` 中使用了 `StartupRecoveryResult`、`diagnose_llm`，但没有完整导入，说明入口本身尚未完全收口。

3. `/chat` 服务契约不稳定。
   `chat_service.py` 当前把普通聊天、意图识别、测试命令、Agent 工作流触发等逻辑混在一起，且返回值结构不统一，容易导致 `response_model` 与真实返回内容不一致。

4. `run_service.py` 责任过重。
   单文件承担了任务生成、执行、修复、日志、恢复、输出格式化等多项职责，后续继续堆功能会放大维护成本。

5. 文档和实现已经发生漂移。
   `backend/README.md`、`docs/backend/api-specification.md` 中有部分能力描述强于当前真实实现，例如消息系统、LangGraph 集成和部分接口契约，需要重新对齐。

6. 测试基线薄弱。
   当前几乎没有有效的后端自动化测试，意味着每次合并或重构都缺少安全网。

## 2. 总体优化原则

本轮后端优化遵循以下顺序：

1. 先稳定运行，再继续扩展功能。
2. 先统一契约，再进行去冗余。
3. 先拆职责边界，再补增强能力。
4. 每完成一阶段，都补最小测试验证。

一句话概括：

> 先把后端从“合并后的半成品状态”收敛成“可启动、可测试、契约一致”的状态，再继续做去冗余和功能增强。

## 3. 分阶段方案

## P0：稳定化修复

目标：让后端重新回到“能启动、能导入、核心接口可用”的状态。

本阶段任务：

1. 统一 `backend.app` 内部导入方式。
   建议全部切换为包内相对导入或标准包导入，保证 `backend.app.main` 能被稳定导入。

2. 修复 `main.py` 中缺失的导入和入口依赖。
   补全 `StartupRecoveryResult`、`diagnose_llm` 等依赖。

3. 收敛 `/chat` 返回契约。
   `generate_chat_response()` 应固定返回统一的数据结构，例如 `ChatServiceResult`，避免有时返回 tuple、有时返回 dataclass。

4. 修复测试命令分支与 `INTENT_TYPE` 的冲突。
   `/test` 不应再返回未定义的 intent 值，需与当前 schema 对齐。

5. 让普通聊天、测试命令、coding 分支在异常情况下都能优雅降级，而不是直接中断接口。

6. 补充最小冒烟测试。
   至少验证：
   - `backend.app.main` 可导入
   - `/health` 可访问
   - `/llm/diagnostics` 可访问
   - `/chat` 在未配置 LLM 时可稳定降级

## P1：接口契约对齐

目标：让代码、schema、README、接口设计文档表达同一件事。

本阶段任务：

1. 对齐以下文件中的接口定义：
   - `backend/app/main.py`
   - `backend/app/schemas.py`
   - `backend/README.md`
   - `docs/backend/api-specification.md`

2. 重新确认当前真实支持的 API 面：
   - `/health`
   - `/llm/diagnostics`
   - `/chat`
   - `/runs`
   - `/runs/summary`
   - `/runs/{run_id}`
   - `/messages`

3. 对“文档里已承诺但代码未稳定支持”的内容做二选一：
   - 真正补全实现
   - 暂时降级为“规划中 / 实验中”

## P2：去冗余与职责拆分

目标：降低单文件复杂度，减少后续继续开发时的耦合。

本阶段任务：

1. 拆分 `chat_service.py`：
   - `intent_detection`
   - `chat_reply`
   - `test_commands`
   - `agent_bridge`

2. 拆分 `run_service.py`：
   - `run_query_service`
   - `run_execution_service`
   - `run_codegen_service`
   - `run_recovery_service`
   - `run_formatters`

3. 收拢重复逻辑：
   - 文本预览
   - 输出裁剪
   - 错误格式化
   - 响应映射

## P3：消息系统与功能完善

目标：把现有消息和 Agent 相关能力从“能演示”推进到“能稳定联调”。

本阶段任务：

1. 强化消息队列：
   - 提升消息 ID 唯一性
   - 明确 `since_id` 找不到时的行为
   - 增加线程安全保护
   - 统一时间戳格式

2. 为 `/messages` 增加结构化 schema。

3. 明确 LangGraph 定位：
   - 如果是实验功能，则与主线接口解耦
   - 如果要进入主线，则需要真正接入 `/chat` 或 `/runs` 生命周期

4. 继续完善 `/runs` 生命周期能力：
   - cancel
   - retry / rerun
   - 更清晰的失败分类
   - artifact 管理

## P4：测试体系补全

目标：为后续重构、联调和合并提供安全网。

本阶段任务：

1. 增加导入与启动测试。
2. 增加 `/chat`、`/llm/diagnostics`、`/messages` 基础接口测试。
3. 增加消息队列单元测试。
4. 增加 `/runs` 任务流基础测试。
5. 增加 safe tools 测试。

## 4. 推荐执行顺序

建议按照以下顺序推进：

1. 先完成 P0：稳定启动链路和 `/chat` 契约
2. 再完成 P1：统一文档与 schema
3. 然后进入 P2：去冗余、拆职责
4. 之后推进 P3：消息系统与 Agent 主线能力
5. 最后补齐 P4：自动化测试护栏

1. 新建本方案文档

3. 修复 `main.py` 入口缺失依赖
4. 收敛 `chat_service.py` 的返回结构
5. 新增最小冒烟测试并执行验证
