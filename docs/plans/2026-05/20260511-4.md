# 2026-05-11 Backend Runtime Contract Alignment Fix Plan

副标题：后端运行契约对齐修复计划 / Backend Runtime Contract Alignment Fix Plan

## 1. 背景

当前后端主线已经切到单 Agent Loop：

```text
/chat
-> Agent Loop
-> perceive
-> plan
-> act
-> observe
-> decide
-> finalize / failure
```

本轮复查依据：

- `docs/backend/agent-acceptance.md`
- `docs/backend/api-specification.md`
- `docs/backend/module-map.md`
- `docs/plans/2026-05/20260511-3.md`
- `docs/design/main-guides/AI Agent 架构优化V2 - Google Docs.md`
- `docs/design/main-guides/AI Agent 开发框架与蓝图 - Google Docs.md`
- `docs/design/main-guides/AI桌宠全栈开发工作流程指南 - Google Docs.md`

复查结论：主路径方向正确，但仍有两类运行契约没有完全闭环。

- 文件已存在时，Agent 能提出“需要确认覆盖”，但用户明确要求覆盖后，实际工具计划仍保持 `overwrite=false`，无法继续完成写入。
- Agent Loop 图外异常，例如图导入失败或线程执行异常，会返回错误响应，但没有补发 `workflow.failed` 终态事件，前端仍可能认为本轮 workflow 未结束。

## 2. 总目标

让后端运行行为与项目文档约定对齐：

- `/chat` 每轮请求都必须有明确结果。
- `/messages` 每轮必须出现 `workflow.completed` 或 `workflow.failed` 终态事件。
- 文件写入默认安全，不静默覆盖；但用户明确确认覆盖时必须能继续执行。
- 历史 mock、旧节点 metadata 等非主线内容不干扰当前 Agent Loop 判断。

## 3. 分阶段计划

### P0：修复文件覆盖确认续执行

状态：

```text
已完成
```

目标：

- 保持默认不覆盖文件。
- 当用户明确说“覆盖”“替换”“overwrite”“replace”等覆盖意图时，把 `workspace.write` 或 `workspace.export_desktop` 的 `overwrite` 设置为 `true`。
- 不把普通“确认执行”误当作文件覆盖确认，避免扩大风险。

范围：

- `backend/app/agent_workflow/loop/agent_loop_graph.py`
- `backend/tests/test_agent_loop_graph.py`
- `backend/tests/test_agent_loop_acceptance.py`

验收标准：

- `请创建 notes/existing.txt，内容是new` 在文件已存在时仍返回确认提示，不覆盖旧内容。
- `请覆盖 notes/existing.txt，内容是new` 能完成写入，旧内容变为 `new`。
- `/chat` 路径也能触发相同行为，并产生 action event 和 terminal event。

### P1：补齐图外异常终态事件

状态：

```text
已完成
```

目标：

- 当 `build_agent_reply()` 在 Agent Loop 导入或执行阶段发生异常时，除了返回错误响应，还要向消息流发送 `workflow.failed`。
- 避免前端因为缺少终态事件而停留在 loading 状态。

范围：

- `backend/app/services/chat_action/agent.py`
- `backend/tests/test_agent_loop_graph.py`

验收标准：

- 图外异常返回 `ok=false`。
- `/messages` 中能看到 `workflow.failed`，状态为 `error`。
- metadata 能说明失败发生在 Agent Loop 外层保护逻辑。

### P2：隔离或清理历史 mock 与旧节点 metadata

状态：

```text
已完成
```

目标：

- 降低组员阅读后端结构时的误判成本。
- 防止 `mock_backend.py`、旧 route 节点命名、旧 node event 配置被误认为当前主线。

范围：

- `backend/app/mock_backend.py`
- `backend/dev/mock_backend.py`
- `backend/app/agent_workflow/contracts/workflow_nodes.py`
- `backend/app/agent_workflow/output/node_events.py`
- 相关文档说明

验收标准：

- 不再让 mock 服务看起来像主后端入口。
- 旧 route 节点 metadata 要么删除，要么明确标注为 legacy/test-only。
- 不破坏现有测试和前端联调入口。

### P3：文档同步与验收收尾

状态：

```text
已完成
```

目标：

- 同步 README、模块说明和验收文档中的运行契约。
- 跑完整后端测试，确认没有破坏现有能力。

验收标准：

- `backend/README.md` 与 `docs/backend/module-map.md` 对 Agent Loop 主线描述一致。
- `docs/backend/agent-acceptance.md` 覆盖文件覆盖确认、终态事件、workspace 写入、桌面导出。
- 后端测试通过。

## 4. 当前执行顺序

本轮先做：

```text
P0 -> P1 -> 相关测试
```

随后完成：

```text
P2 -> P3
```
