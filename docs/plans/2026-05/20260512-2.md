# 文件工具体验修复方案 / File Tool UX Repair Plan

## 背景 / Background

当前后端已经基本完成从旧路由模式向 `Turn Controller + workflow subgraph` 的方向收敛，但桌宠在文件相关任务上的可用性仍然不足。用户一旦要求创建、读取、重命名、删除、复制或搜索文件，体验容易表现为能力缺失、状态不清晰或动作无法闭环。

本方案只围绕文件能力补齐，不改变整体智能体方向：

```text
/chat
-> Turn Controller
-> coding/file action planning
-> workspace tools
-> Bridge JSON status/action events
-> frontend visible result
```

## 目标 / Goals

1. 让常见文件任务成为稳定的一等能力，而不是依赖零散关键词和临时补丁。
2. 文件工具必须继续受 workspace 边界约束，不能访问任意系统路径。
3. 高风险动作必须保留确认或安全限制，尤其是删除、桌面导出、覆盖。
4. 每次工具执行都要有明确结果摘要，避免前端表现为“卡住没动静”。
5. 补齐测试，把文件任务作为后端验收重点。

## 非目标 / Non-goals

1. 本阶段不接入任意 shell 文件命令。
2. 本阶段不允许绕过 `workspace` 和显式配置的桌面导出目录。
3. 本阶段不做复杂多文件项目级重构智能体，只补齐桌宠高频文件动作。
4. 本阶段不把前端升级成完整文件管理器，只要求结果可读、状态清晰。

## P1：补齐 Workspace 文件动作 / Add Workspace File Actions

新增工具：

```text
move_workspace_path
copy_workspace_path
delete_workspace_path
search_workspace_text
```

对应 action：

```text
workspace.move
workspace.copy
workspace.delete
workspace.search
```

安全策略：

1. `workspace.move` 和 `workspace.copy` 只允许 workspace 内路径。
2. `workspace.delete` 默认仅允许删除文件；目录删除必须显式 `recursive=true`，并在 agent action 层标记为高风险确认动作。
3. `workspace.search` 只读取 workspace 内文本文件，并裁剪结果数量和预览长度。
4. 所有路径都继续通过 `resolve_workspace_path` 校验边界。

## P2：增强自然语言文件计划 / Improve File Planning

优先支持这些输入形式：

```text
请创建 notes/demo.md，内容是 # Demo
请读取 notes/demo.md
请列出 notes 目录结构
请把 notes/a.txt 改名为 notes/b.txt
请把 notes/a.txt 复制到 notes/copy.txt
请删除 notes/copy.txt
请查找 notes 下包含 hello 的文件
```

规划规则：

1. 显式路径优先。
2. 两个路径同时出现时，按源路径、目标路径解析移动/复制。
3. 只有一个目标文件名时，重命名默认沿用源文件所在目录。
4. 搜索必须有查询词；没有查询词时不执行搜索。
5. 不确定的复杂请求继续退回 `run.create`，避免假装能做。

## P3：状态与结果反馈 / Status And Result Feedback

当前 Bridge JSON 事件已经存在，后续文件动作应继续复用：

```json
{
  "type": "Status_Update",
  "payload": {
    "node": "act_node",
    "action": "workspace.search",
    "quip": "正在搜索文件内容..."
  }
}
```

本阶段先保证 action started/completed/failed 事件含有正确 `action_name`、工具结果摘要、文件目标字段和简短 quip 文案；更完整的卡片式展示可以后续前端联调时集中处理。
