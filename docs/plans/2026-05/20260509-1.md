# 2026-05-09 后端方向对齐与纠偏计划

## 1. 依据文档

本次计划直接依据以下 3 份设计文档与现有项目计划整理：

1. `docs/design/main-guides/AI Agent 架构优化V2 - Google Docs.md`
2. `docs/design/main-guides/AI Agent 开发框架与蓝图 - Google Docs.md`
3. `docs/design/main-guides/AI桌宠全栈开发工作流程指南 - Google Docs.md`
4. `docs/plans/2026-05/20260506-2.md`

这 3 份设计文档反复强调的核心结论是一致的：

1. 项目目标是 **AI 桌宠**，不是泛化成臃肿通用平台。
2. `LongCat` 或其他大模型只是 `LLM provider`，不是 Agent 本体。
3. `LangGraph` 才是 Agent Brain，负责状态流转、节点编排、工具调用。
4. 前台角色表现状态与后台工程状态必须隔离，避免上下文腐化。
5. 自然语言主链路不能被脆弱的关键词规则轻易拦死。
6. 本地工具能力必须继续受安全边界约束。

## 2. 当前后端的主要偏航点

### 2.1 偏航点 A：意图路由过度依赖关键词

当前 `detect_intent()` 主要依赖硬编码关键词。

直接后果：

1. `你是谁` 能进入 `chat`，但 `我是谁` 这类正常自然语言可能落入 `unknown`。
2. 项目虽然已经接入 `LangGraph router -> chat_node -> roleplay_node`，但很多普通聊天在进入图之前就被规则层拦住了。
3. 这和设计文档中“Intent Router 是纯分类器，而不是脆弱关键词门槛”的方向不一致。

### 2.2 偏航点 B：默认人格提示词仍偏通用助手

当前默认 `LLM_SYSTEM_PROMPT` 虽然提到 Live2D 桌宠，但整体更像“通用中文助手”。

直接后果：

1. 角色感不稳定。
2. 更容易把工程噪声直接回给用户。
3. 和设计文档中“前台看板娘 / Roleplay Agent”定位不完全一致。

### 2.3 偏航点 C：`unknown` 收口过于生硬

当前 `unknown` 分支文案偏“请重新分类输入”，更像调试型接口，而不是桌宠前台角色。

直接后果：

1. 用户体验容易断掉。
2. 即使走到兜底分支，收口方式也不够像桌宠。

### 2.4 偏航点 D：前端尚未正式复用 `session_id`

这是当前链路里最明显的跨端缺口之一：

1. 后端已经实现 `session_id + conversation_store`。
2. Electron 当前仍主要发送 `{prompt, context}`，未把后端返回的 `session_id` 正式串回下一轮请求。

说明：

- 这一项不是纯后端能单独彻底修好的问题。
- 本轮记录为跨组协作依赖，不在本次“后端单边纠偏”内强行解决。

## 3. 本轮纠偏目标

本轮只做 **顺着项目方向、且后端单边能稳定落地** 的修正：

1. 让普通自然语言默认更容易进入 `chat`，而不是被错误打进 `unknown`。
2. 保留 `unknown` 分支，但只让它处理真正模糊、符号化、不可理解的输入。
3. 把默认系统提示词向“桌宠前台角色 + 简洁工程收口”方向收紧。
4. 保持 `LangGraph`、`/runs`、`safe tools`、`messages` 这条现有主链路不被破坏。

## 4. 本轮具体改动

### 4.1 已执行：自然语言聊天判定纠偏

处理思路：

1. 继续优先保留强确定性的 `coding` 判定。
2. 对普通自然语言增加更稳妥的聊天识别逻辑。
3. 将 `unknown` 保留给符号串、纯噪声或高度不明确输入。

目标效果：

1. `我是谁`
2. `你还记得我刚才说了什么`
3. `random words without known keywords`

这类输入应优先走 `chat`，而不是被错误拦下。

### 4.2 已执行：默认角色提示词收紧

处理思路：

1. 明确“桌宠前台看板娘”定位。
2. 保留自然中文与轻微角色感。
3. 明确禁止把未确认的执行结果说成已完成。
4. 明确要求优先输出简短总结，而不是直接倾倒原始工程噪声。

### 4.3 已执行：`unknown` 收口文案柔化

处理思路：

1. 不再要求用户强行先做“聊天 / 代码任务”分类。
2. 改成可继续聊天、也可补充任务信息的低阻塞兜底文案。

## 5. 本轮之后仍要继续推进的内容

### 5.1 需要前端配合的事项

1. 正式保存并复用后端返回的 `session_id`
2. 不再只依赖窗口本地拼接的 `context`
3. 逐步切到后端会话主导的多轮记忆链路

### 5.2 仍属于后端下一阶段的事项

1. 将 Intent Router 继续升级为更稳定的分类器，可考虑 LLM 分类 + 规则兜底
2. 逐步让 `Roleplay Node` 从“当前的统一收口层”演进为更明确的角色表达层
3. 继续收紧“前台展示状态”和“后台工程状态”的边界
4. 按设计文档继续推进 QA / Debugger / Subgraph / HITL 等后续能力

## 6. 当前明确不做的事

为了不漂，本轮不做以下事情：

1. 不把项目扩成完整多 Agent 公司化系统
2. 不引入用户系统、数据库、权限系统
3. 不把后端重写成复杂通用平台
4. 不为了解决聊天识别问题而大改 `/runs` 主链路

## 7. 结论

本轮纠偏的原则是：

1. **修正错误的拦截**
2. **保留已有主链路**
3. **向桌宠前台角色靠拢**
4. **不脱离 LangGraph + Tools + Safety 的总方向**

一句话总结：

> 后端下一步不是“功能越多越好”，而是“让自然对话、角色表现、LangGraph 编排和安全工具这四条线更一致”。
