# 真实项目访问集成方案 / Real Project Access Integration Plan

## Goal / 目标

把 `project-access` 分支中的真实项目访问能力安全集成到当前 `main` 架构中，同时避免旧分支覆盖现有 Agent loop、文件 workflow、富文本协议和文档结构。

## Background / 背景

当前主线已经完成：

- Agent loop 作为默认 `/chat` 运行路径。
- 文件任务 workflow、文件上下文和 workspace tools 拆分。
- assistant Markdown / 代码块 / LaTeX 富文本协议。
- `docs/` 目录重组和 `scripts/ai/*` 验证脚本。

`project-access` 分支提供了另一个有价值的能力：让文件工具从默认 `backend/workspace/` 扩展到用户配置的真实项目目录。该分支从较早提交分叉，不能直接替代主线，只能选择性集成核心安全文件层。

## Integrated Scope / 已集成范围

集成方式：

- 从 `main` 新建 `integrate/project-access-on-main`。
- 只 cherry-pick 真实项目访问相关的核心代码提交。
- 不直接 merge 旧分支，避免覆盖当前主线结构。

当前纳入文件：

```text
backend/.env.example
backend/app/core/config.py
backend/app/tools/safe_fs.py
backend/tests/tools/test_safe_fs.py
backend/README.md
docs/backend/module-map.md
```

核心能力：

- `PROJECT_ROOT`：配置真实项目根目录；为空时继续使用 `backend/workspace/`。
- `PROJECT_WRITE_ENABLED`：控制真实项目写入权限，默认 `false`。
- `get_effective_workspace_dir()`：统一返回当前文件工具根目录。
- `is_excluded_path()`：拒绝访问敏感目录和文件。
- `check_write_permission()`：真实项目只读模式下阻止写入。
- `resolve_workspace_path()`：统一做根目录范围检查和排除规则检查。
- `safe_write_file()`：写入前强制检查权限。

## Safety Rules / 安全规则

- 不修改 `backend/.env`。
- 不把真实项目访问能力接成任意磁盘访问。
- 不允许绕过 `safe_fs.py` 直接读写用户项目。
- 不允许访问 `.git`、`.env*`、`node_modules`、`__pycache__`、`.venv`、`dist`、`build` 等敏感或生成路径。
- 即使 `PROJECT_WRITE_ENABLED=true`，Agent 层仍应保留高风险文件写入、删除、覆盖确认。
- 真实项目访问只解决“工具根目录”问题，不等于允许 shell、Git、依赖安装或外部命令自由执行。

## Remaining Work / 后续工作

- 增加项目访问 smoke 测试：临时 `PROJECT_ROOT` 下读取普通文件、拒绝 `.env`、拒绝越界路径、只读模式拒绝写入。
- 在 Agent 文件 workflow 中明确区分 `workspace` 与 `project` 的用户可见文案，避免用户误以为写到了错误位置。
- 后续如需支持真实项目命令执行，应单独制定 shell / Git 权限方案，不应复用本次文件访问开关直接放行。
- 可考虑新增更细粒度配置，例如 `PROJECT_ACCESS_MODE=workspace/project_readonly/project_write`，减少布尔开关带来的误解。
