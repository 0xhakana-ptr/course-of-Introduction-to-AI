# 文件工作流与富文本协议优化方案 / File Workflow And Rich Text Protocol Plan

## Goal / 目标

把当前“单工具式文件能力”升级为更接近 Agent 的文件任务工作流，并为富文本输出建立明确协议，减少前端靠内容猜测渲染方式的问题。

```text
设计文档
-> 任务分解
-> 代码生成
-> 测试验证
-> 迭代优化
```

## Current Problems / 当前问题

已完成的 `20260512-2` 主要补齐了工具箱：

```text
workspace.write/read/list/move/copy/delete/search/export_desktop
```

但用户体验仍然不稳定，原因是：

1. 文件任务缺少专门 workflow，仍主要靠 `plan_workspace_tool()` 规则直连单个 action。
2. 缺少文件上下文记忆，无法稳定理解“刚才那个文件”“上一个搜索结果”等指代。
3. 多行内容、Markdown、代码块、LaTeX、JSON 的写入提取仍不够可靠。
4. 富文本渲染依赖消息类型和前端猜测，后端没有统一 `content_type/render_mode` 协议。
5. 多步文件任务缺少 `plan -> execute -> observe -> repair -> summarize` 的闭环。

## Architecture Direction / 架构方向

目标路径：

```text
/chat
-> Turn Controller
-> file workflow subgraph
-> workspace actions
-> observer
-> repair or finish
-> Bridge JSON + rich text protocol
```

关键原则：

1. `agent_loop_graph.py` 只负责识别和派发，不继续增长为文件大脑。
2. 文件任务逐步进入 `agent_workflow/file/` 子图。
3. 简单文件动作仍可复用现有 workspace tools。
4. 多步、上下文相关、内容复杂的文件任务由 file workflow 接管。
5. 富文本渲染应由协议字段驱动，而不是前端猜。

## Task Breakdown / 任务拆分

### P0: 建立 file workflow 骨架

新增：

```text
backend/app/agent_workflow/file/
  __init__.py
  file_graph.py
  state.py
  result.py
```

目标：

1. 先不大改行为。
2. 让简单 workspace action 可以通过 file workflow 包装执行。
3. 为后续文件上下文、观察、修复留接口。

验收：

1. 现有 workspace 文件任务继续通过。
2. 新增 file workflow 单元测试。

状态：

已完成。新增 `backend/app/agent_workflow/file/` 子图骨架，支持通过统一 workflow result 包装现有 workspace action，并完成聚焦测试验证。

### P1: 文件上下文记忆

新增轻量状态：

```text
last_created_file
last_read_file
last_written_file
last_deleted_file
last_search_results
last_file_action
```

目标：

支持：

```text
读取刚才创建的文件
把刚才那个文件复制一份
删除刚才搜索到的文件
```

状态：

已完成。新增按 `session_id` 保存的文件上下文，workspace action 成功后记录最近创建、读取、写入、删除和搜索结果；agent loop 规划阶段会用这些信息解析“刚才创建的文件”“刚才那个文件”“刚才搜索到的文件”等指代。

### P2: 多行内容提取

增强写文件内容解析：

```text
内容是：
```python
print("hello")
```

内容如下：
# Title

$$
\int \frac{1}{x} dx
$$
```

目标：

1. 保留代码块和 Markdown。
2. 不把“然后读出来”“再运行一下”等后续动作写入文件内容。
3. 覆盖 Python/C++/bash/Markdown/LaTeX/JSON。

状态：

已完成。写文件解析器现在支持 `内容如下`、`代码如下` 等多行输入；会保护 fenced code block 和 LaTeX block，不会把块内的“然后/then”等词误判为后续动作；写入代码文件时会剥离最外层代码围栏，写入 Markdown 时保留代码块和公式块。

### P3: 富文本协议

新增或扩展消息字段：

```json
{
  "content_type": "markdown",
  "render_mode": "rich_text"
}
```

目标：

1. assistant markdown 明确标记。
2. 文件读取结果可明确标记为 `plain_text` 或 `markdown`。
3. 状态消息仍保持轻量 `plain_text`。
4. 前端根据协议渲染，不再靠消息来源猜。

状态：

已完成。`/chat` 响应、chat 消息和 bridge roleplay payload 均显式携带 `content_type=markdown`、`render_mode=rich_text`；前端根据 `render_mode` 决定是否用 Markdown/KaTeX/highlight.js 渲染，并保留默认兼容。

### P4: 多步文件任务闭环

支持：

```text
创建 demo.py，写入代码，然后运行测试
创建 README.md，写入 Markdown，然后读出来确认
搜索 hello，把第一个结果复制到 backup.txt
```

目标：

```text
plan
-> execute
-> observe
-> repair or continue
-> summarize
```

状态：

已完成。agent loop 现在支持 search action 成功后的动态后续队列：当用户要求“搜索/查找 ... 然后处理第一个结果”时，会先执行 search，再在 observe 阶段基于首个搜索结果排队 read/copy/delete 等后续 workspace action，避免旧规则把“搜索目录 + 复制目标”误判成直接复制目录。

### P5: 验收与文档

新增验收用例：

```text
多行 Markdown 写入
代码块写入
LaTeX 写入
刚才那个文件
搜索结果指代
文件动作富文本协议
```

同步：

```text
docs/backend/agent-acceptance.md
docs/backend/module-map.md
```

状态：

已完成。已同步 `docs/backend/agent-acceptance.md`、`docs/backend/module-map.md` 和 `backend/README.md`，补充文件上下文、多行内容写入、搜索结果多步闭环、富文本协议和对应测试入口说明。

## Risk Gate / 风险门

本方案可自动执行的低风险项：

1. 新增 file workflow 骨架。
2. 包装已有 workspace action。
3. 补测试。
4. 新增协议字段但保持向后兼容。
5. 文档同步。

需要用户确认的事项：

1. 移除旧 `plan_workspace_tool()` 路径。
2. 改变公开 API 返回格式。
3. 移除旧前端消息字段。
4. 引入大型富文本编辑器依赖。
5. 允许更高风险文件操作。
