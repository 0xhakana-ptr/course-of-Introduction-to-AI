# 2026-05-11 Agent Loop Runtime Hardening Plan

## 1. 背景

旧 route graph、`AGENT_RUNTIME_MODE` 灰度开关和 legacy route diagnostics 已经移除。

当前后端主路径变为：

```text
/chat
-> Agent Loop
-> perceive
-> plan
-> act
-> observe
-> decide
-> finalize / failure
```

后续优化不再围绕“route/loop 两套模式如何共存”展开，而是围绕 Agent Loop 本身的真实运行能力、工具安全边界、消息反馈和桌宠联调体验继续完善。

## 2. 总目标

让后端从“能跑 Agent Loop”推进到“真实桌宠 Agent 可稳定使用”。

核心目标：

- `/chat` 每次请求都能进入明确终态，不让前端长期 loading。
- 工具动作有清晰状态反馈，用户能知道当前在理解、规划、执行、观察还是完成。
- 文件、workspace、桌面导出和 run 控制动作有明确安全边界。
- Agent Loop 能处理简单多步任务，而不是只做一次 action 后结束。
- 普通聊天和短问题不消耗过多 token。
- 文档和验收流程能让组员稳定复现、联调和排查问题。

## 3. 分阶段计划

### P1：功能层验收基线

目标：

- 先确认当前 Agent Loop 主路径在真实桌宠里不会卡住。
- 建立一份可重复执行的功能验收清单。

范围：

- 普通聊天：`你是谁`、`1+1=?`、`我是谁`。
- workspace 文件操作：创建 txt、读取 txt、列目录。
- run 任务：创建任务、查看 run 状态、取消任务、retry、rerun。
- 桌面导出：默认禁止、需要确认、配置后可导出。
- 消息流：quip、status、action、completed、failed。

验收标准：

- `/chat` 每次都有返回。
- `/messages` 每次都有 `workflow.completed` 或 `workflow.failed`。
- 前端输入框不会因为后端没有终态事件而锁死。
- “创建文件”类任务必须返回成功、失败、禁止原因或确认请求之一。

优先级：

```text
最高
```

原因：

- 当前最需要先验证的是用户真实输入是否会卡住。
- 这一步能直接暴露前后端联调、消息协议、工具执行和终态事件的问题。

### P2：Agent Loop 多步循环能力

目标：

- 强化 `observe -> decide -> plan` 的再规划能力。
- 让 Agent Loop 支持简单多步任务。

范围：

- 统一 `observe` 输出结构。
- 让 `decide` 判断是否需要继续下一步，而不是多数场景直接结束。
- 明确 `max_steps`、`stop_reason`、`recovery_reason`。
- 支持“创建文件后再读取确认内容”这类简单多步任务。
- 工具失败后能进入降级动作或最终说明。

验收标准：

- “创建 notes/a.txt，内容 hello，然后读出来确认”能形成至少两步 action。
- 工具失败不会静默卡死。
- 超出最大步数时有明确 stop reason。
- 每一步都有 runtime event 和用户可理解的状态反馈。

优先级：

```text
中高
```

原因：

- 多步循环是 Agent 区别于固定路由的关键。
- 但必须在 P1 证明基础链路稳定后再推进。

### P3：动作安全与确认机制

目标：

- 对有副作用的动作建立统一确认机制。
- 避免桌宠误覆盖文件、误导出桌面、误取消任务。

范围：

- 给 action 增加统一风险等级：低、中、高。
- 对高风险动作统一走 confirmation。
- 覆盖文件已存在、覆盖写入、桌面导出、执行测试、run cancel。
- 确认消息包含 action 名称、目标路径、风险说明和下一步。
- 用户确认后能继续执行，不确认则优雅结束。

验收标准：

- 已存在文件不会被直接覆盖。
- 桌面导出默认不会直接写桌面。
- 用户确认后能继续执行对应动作。
- diagnostics 能看出动作是否需要确认。

优先级：

```text
高
```

原因：

- 桌宠 Agent 会调用工具，安全边界不能依赖口头约定。
- 这部分直接影响项目是否可靠。

### P4：Workspace 和文件工具完善

目标：

- 优先打磨最接近用户价值的文件能力。
- 让“创建、读取、列目录、导出”形成稳定体验。

范围：

- 优化中文路径、中文标点、空格路径解析。
- 稳定支持 txt、md、json、py 等常见文本文件。
- 统一文件写入返回格式。
- 区分 workspace 创建和桌面导出。
- 文件路径错误时给出可执行建议。

验收标准：

- “在 workspace 创建 txt 文件”可以直接完成。
- “在桌面创建 txt 文件”不会卡住，必须返回确认、禁止原因或成功结果之一。
- 文件已存在时明确提示，不直接覆盖。
- 文件读取结果包含路径、内容摘要和是否被截断。

优先级：

```text
高
```

原因：

- 文件操作是桌宠 Agent 最直观的工具能力。
- 之前“创建桌面 txt 卡住”的问题应在这一阶段彻底验收。

### P5：记忆、上下文和 token 控制

目标：

- 降低普通对话和短任务的 token 消耗。
- 避免简单问题带入过多工程上下文。

范围：

- 检查 `/chat` 每次传给 LLM 的上下文长度。
- 优化 conversation summary 触发条件。
- 普通短问题只带必要会话上下文。
- 工具任务只传必要工具结果，不传完整 trace 和长日志。
- diagnostics、run output、错误详情进入 prompt 前要摘要化。

验收标准：

- 简单聊天 token 明显下降。
- 连续对话仍保留必要记忆。
- 长日志不会直接灌入 LLM prompt。
- LLM 失败时仍有本地 fallback 说明。

优先级：

```text
中
```

原因：

- token 成本和响应速度会影响体验。
- 但要在功能链路稳定后再做，否则容易优化错对象。

### P6：文档和开发体验收尾

目标：

- 文档进入单 Agent Loop 主线状态。
- 降低组员启动、联调和排查成本。

范围：

- 更新 README 启动、测试、联调步骤。
- 更新 backend module map。
- 更新 API 文档中 `/chat`、`/messages`、`/agent/diagnostics/*` 的语义。
- 写“桌宠 Agent 功能验收手册”。
- 写常见问题排查：LLM 未接通、WebSocket 断开、文件写入失败、桌面导出失败、前端 loading。

验收标准：

- 新组员按 README 能启动后端并完成基础验收。
- 队长提到的 quip、节点 JSON、动作反馈能在文档中找到对应实现位置。
- 文档不再出现 route/loop 二选一的误导性说明。

优先级：

```text
中
```

原因：

- route 已经删除，文档必须收敛到单主线。
- 但文档收尾应跟着 P1-P5 的实际落地结果更新。

## 4. 推荐执行顺序

建议顺序：

```text
P1 功能层验收基线
-> P3 动作安全与确认机制
-> P4 Workspace 和文件工具完善
-> P2 Agent Loop 多步循环能力
-> P5 记忆、上下文和 token 控制
-> P6 文档和开发体验收尾
```

理由：

- 先解决真实运行是否卡住的问题。
- 再收紧有副作用工具的安全边界。
- 然后打磨用户最容易感知的文件能力。
- 多步循环应建立在稳定的 action 和 observe 基础上。
- token 控制和文档收尾放在功能行为稳定之后更准确。
