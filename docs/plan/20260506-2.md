# 2026-05-06 AI 桌宠后端后续完善计划

## 1. 项目目标判断

当前项目不是普通 Web 后台，而是一个面向 **Live2D AI 桌宠** 的本地后端。

因此后端的核心目标不是堆叠复杂企业级能力，而是稳定支撑以下几件事：

1. 让聊天窗口可以接入真实 AI。
2. 让 AI 回复具备稳定的人格和上下文。
3. 让 AI 能通过消息协议驱动桌宠的字幕、表情、动作和状态。
4. 让 LangGraph 作为 Agent Brain 组织状态流、节点和工具调用。
5. 让代码任务或工具任务可以安全执行，并把执行过程反馈给前端。
6. 保持本地桌面项目的轻量、可调试和可协作。

一句话概括：

> 后端应该成为桌宠的“大脑”和“行为调度层”，而不是泛化成一个臃肿的通用平台。

这里需要明确一个核心边界：

1. `LongCat` 是大模型本身，是 LLM provider，负责推理、生成、判断和写作。
2. `LangGraph` 是 Agent Brain 的编排框架，负责状态流转、条件分支、节点协作和工具调用。
3. `safe_fs`、`safe_execute_command`、`/runs`、`character_interface` 是工具和能力层。
4. 真正的 Agent 不是单独的 LongCat，而是 `LangGraph + LLM + Tools + State + Safety` 的组合。

## 2. 当前后端状态

当前后端已经具备较完整的基础骨架：

1. `FastAPI` 主入口可以正常启动。
2. `/chat` 已经可以处理普通聊天、测试命令和实验性 coding 分支。
3. `/runs` 已经支持任务创建、执行、自动修复、重试、重新运行、取消、日志查询和 attempts 查询。
4. `/messages` 已经支持统一消息队列和前端轮询。
5. `/llm/diagnostics` 已经支持本地配置检查和远程连通性诊断。
6. `safe_fs` 和 `safe_execute_command` 已经提供 workspace 边界和危险命令拦截。
7. 后端自动化测试已经覆盖启动、聊天、LLM、消息队列、runs 生命周期和 safe tools。
8. `agent_workflow` 已升级为最小 LangGraph Agent Brain，当前 coding intent 已能通过 Run Tool Node 创建 run；完整 QA、Repair 和多 Agent 协同仍后置。

当前后端已经不是“能不能跑”的阶段，而是进入了“是否贴合桌宠产品目标”的阶段。

## 3. 当前还缺什么

### 3.1 缺 AI 回复到角色表现的稳定链路

现在后端可以返回聊天文本，也可以通过消息队列发送 `quip`、`expression`、`chat`、`status`、`error`。

但还缺一个统一策略来决定：

1. AI 什么时候只回复聊天文本。
2. AI 什么时候发字幕窗口的打趣话语。
3. AI 什么时候切换表情。
4. AI 什么时候播放动作。
5. AI 什么时候发送任务进度状态。

这是 AI 桌宠项目最关键的缺口。用户感受到的不是“后端有接口”，而是“角色真的在思考、回应和表现情绪”。

### 3.2 缺统一的角色人格与系统提示词

当前 `.env.example` 中有 `LLM_SYSTEM_PROMPT`，但它仍然偏通用助手。

后续需要明确：

1. 桌宠的人格设定。
2. 回复语气。
3. 专业回答和轻松吐槽的边界。
4. 长回答和短回答的策略。
5. 是否允许主动追问。
6. 是否允许通过消息队列触发表情或动作。

这部分会直接决定项目体验，比继续增加后端接口更重要。

### 3.3 缺聊天上下文管理

当前 `ChatRequest` 虽然有 `context` 字段，但后端自身没有维护会话历史。

这意味着：

1. 如果前端不传完整上下文，后端就不会记得前面说过什么。
2. 多轮聊天体验会偏割裂。
3. 后续想做“桌宠记忆”会缺基础结构。

建议先做轻量版本：

1. `session_id`
2. 最近 N 条消息
3. 清空会话
4. 可选摘要

### 3.4 缺后端到 Live2D 行为的正式适配层

前端已经提供 Live2D WebSocket 控制接口，并支持表情标签映射，例如 `tag:开心`。

后端现在还没有一个正式服务来表达：

1. `thinking` 应该对应什么字幕和表情。
2. `done` 应该对应什么表情和动作。
3. `error` 应该对应什么表情和状态。
4. `coding` 应该对应什么状态提示。

建议新增一个角色事件服务，例如：

```text
backend/app/services/character_action/
backend/app/services/character_interface.py
```

它不需要直接绑定具体 Live2D 文件名，而是优先输出通用语义，例如：

```text
tag:开心
tag:疑惑
tag:思考
tag:难过
```

### 3.5 需要继续完善 coding 分支和 `/runs` 的统一边界

当前 `/chat` 的 `coding` 分支已经进入最小 LangGraph Agent Brain，并由 Run Tool Node 复用现有 run service 创建任务。

后续还需要继续补齐：

1. 将普通 chat intent 也逐步纳入 LangGraph，同时不破坏现有会话记忆。
2. 让 LangGraph 能读取 run 执行结果，而不是只负责创建 run。
3. 增加 QA / Repair Node，用于分析失败日志并决定是否触发修复。
4. 让 Roleplay Node 更稳定地根据 run 状态发送字幕、表情和状态。
5. 保持 `/messages` 持续发送任务状态、表情和最终结果。

这样可以继续避免 `coding` 分支和 `/runs` 各做一套任务执行，同时也符合“LangGraph 才是 Agent Brain，大模型只是脑子”的团队方向。

### 3.6 缺更完整的 LangGraph Agent Brain

当前后端已经有最小 LangGraph 图，但还没有形成完整的“规划、执行、读取结果、分析、修复、总结”闭环。

下一步需要补或继续细化的是：

1. Intent Router Node：继续提高意图分类准确率，但不保留复杂历史。
2. Chat Node：逐步承接普通聊天，同时复用现有 LLM client 和 conversation store。
3. Roleplay Node：只负责桌宠人格、字幕、表情和状态，不读取后台工程噪声。
4. Coding Node：把用户需求转成更结构化的可执行任务。
5. Run Tool Node：从“创建 run”升级为“创建、读取、总结 run”。
6. QA / Repair Node：读取执行结果、错误和日志，再决定是否修复。
7. End Node：汇总结果并通知前端。

LangGraph 的价值不在于替代 LongCat，而在于把这些节点和状态组织起来，让大模型在正确的位置发挥作用。

### 3.7 缺更清晰的前后端消息协议最终版

当前 `MESSAGE_TYPE` 有：

1. `quip`
2. `expression`
3. `chat`
4. `error`
5. `status`

但项目本身还支持 Live2D motion，因此后续需要决定：

1. 是否新增 `motion`。
2. 是否把 `expression` 和 `motion` 合并为 `live2d_action`。
3. `expression` 字段到底传通用标签还是真实 action id。
4. `quip` 和 `chat` 的使用边界。
5. `status.progress` 前端是否展示。

这个协议应尽早写进文档，方便前端同学稳定接入。

## 4. 当前不太需要什么

### 4.1 暂时不需要完整多 Agent 公司化协同

当前 `agent_workflow` 已经从线性演示流程收敛为最小 LangGraph 图，但还没有真正形成完整的规划、执行、分析闭环。

建议：

1. 保留 LangGraph 作为主线方向。
2. 暂时不急着做 PM、Architect、Coder、QA 全量多 Agent 公司。
3. 先做一个最小可用 LangGraph：`router -> chat/coding -> roleplay -> done`。
4. 等最小图稳定后，再扩展 PM、Coder、QA、Repair 等节点。

也就是说：不是不要 LangGraph，而是先做小图，不要一上来就做完整多 Agent 公司。

### 4.2 暂时不需要用户系统和数据库

这是本地桌面项目，目前没有多人协作、账号登录、远程权限隔离需求。

短期不建议加入：

1. 用户注册登录。
2. 权限系统。
3. 复杂数据库。
4. ORM 层。

当前文件存储 runs 结果已经足够支撑开发和课程展示。

### 4.3 暂时不需要多厂商 SDK

当前 OpenAI-compatible 主模型和 fallback 模型已经足够，LongCat 可以作为当前主要大模型。

不建议现在引入大量厂商专用 SDK。只要 provider 支持 OpenAI-compatible `/chat/completions`，就继续走同一套 LLM client。

### 4.4 暂时不需要把 `/runs` 全部功能一次性接给前端

`retry`、`rerun`、`cancel`、`attempts`、`logs` 对后端调试很有价值，但前端第一阶段不一定都要展示。

前端最小接入可以先只使用：

1. 创建任务。
2. 查询状态。
3. 展示最终结果。
4. 接收状态消息。

### 4.5 暂时不需要保留正式主线外的 mock 入口

`backend/app/mock_backend.py` 更像早期调试工具，和正式 `/messages` 链路有重复。

建议后续二选一：

1. 如果仍有调试价值，移动到 `backend/devtools/` 或 `scripts/`。
2. 如果前端已经不依赖它，删除或归档。

## 5. 优化方向

### 5.1 路由拆分

当前 `main.py` 同时承担多组接口路由，后续建议拆分为：

1. `backend/app/api/health_routes.py`
2. `backend/app/api/llm_routes.py`
3. `backend/app/api/chat_routes.py`
4. `backend/app/api/run_routes.py`
5. `backend/app/api/message_routes.py`

拆分后 `main.py` 只负责：

1. 创建 FastAPI app。
2. 注册 lifespan。
3. 注册 routers。
4. 配置 logging。

### 5.2 统一角色事件服务

建议新增角色事件层，封装桌宠表现逻辑。

示例能力：

1. `send_thinking()`
2. `send_chatting()`
3. `send_task_started()`
4. `send_task_done()`
5. `send_task_failed()`
6. `send_cancelled()`

这样 `/chat`、`/runs` 和未来 Agent 工作流都可以复用同一套表现策略。

### 5.3 聊天记忆

建议新增轻量会话存储：

```text
backend/app/storage/conversation_store.py
```

第一版不需要复杂，只需要：

1. 创建或复用 `session_id`。
2. 保存最近若干轮对话。
3. 限制最大长度。
4. 支持清空会话。
5. 可选持久化到 workspace。

### 5.4 coding intent 接入 `/runs`

后续应逐步把 `chat_action/coding.py` 从直接桥接实验性 `agent_workflow`，调整为进入 LangGraph Agent Brain。

建议行为：

1. `/chat` 调用 LangGraph 入口。
2. Intent Router 判断 `coding`。
3. Coding Node 准备任务输入。
4. Run Tool Node 调用现有 run service 创建 run。
5. 后台执行 run。
6. Roleplay Node 根据 run 状态发送角色事件。
7. End Node 返回 `run_id` 和简短说明。

### 5.5 LangGraph Agent Brain 最小化改造

建议把 `agent_workflow` 升级为正式的 Agent Brain。

第一版只需要最小图：

```text
router -> chat_node -> roleplay_node -> done
router -> coding_node -> run_tool_node -> roleplay_node -> done
router -> unknown_node -> roleplay_node -> done
```

第一版状态建议：

```text
user_input
session_id
intent
chat_output
run_id
ui_status
error
```

状态设计原则：

1. 不把所有历史都塞进一个全局 `messages`。
2. 聊天上下文走 `conversation_store`。
3. 代码任务上下文走 `/runs` 和 artifacts。
4. 前台角色表现只读 `intent`、`ui_status`、`run_id`、`error` 等摘要信息。
5. 后台工程错误不要污染 Roleplay Agent 的长上下文。

### 5.6 整理实验模块

建议明确区分：

1. 正式主线：`/chat`、`/runs`、`/messages`、LLM client、safe tools。
2. Agent 主线：`agent_workflow`，逐步升级为 LangGraph Agent Brain。
3. 调试工具：`mock_backend.py`。

这样团队协作时不容易把实验代码误认为正式入口。

### 5.7 schema 和内部状态继续收敛

建议遵循：

1. 机器读的字段用稳定英文枚举。
2. 用户可见的 `output`、`summary`、`content` 用中文。
3. 文档中明确每个消息字段的含义。
4. 不让前端依赖临时字符串判断状态。

## 6. 分阶段计划

### P0：协议收口

目标：先把桌宠表现协议定下来。

任务：

1. 整理 `/messages` 的正式消息协议。
2. 判断是否新增 `motion` 或 `live2d_action`。
3. 明确 `quip`、`chat`、`expression`、`status` 的边界。
4. 更新 `docs/backend-api-specification.md`。
5. 补充消息 schema 测试。

产出：

1. 前后端都能依据同一份消息协议联调。
2. 后端不再随意拼临时消息结构。

### P1：角色事件服务

目标：让后端可以稳定驱动桌宠表现。

任务：

1. 新增 `character_interface.py`。
2. 新增 `character_action/`。
3. 封装常见事件：思考、回复、任务开始、任务完成、任务失败、取消。
4. 将 `/chat` 和 `/runs` 中的状态反馈逐步接入角色事件服务。

产出：

1. 角色表现逻辑从业务流程中抽离。
2. 前端能更稳定地收到字幕、表情和状态。

### P2：聊天人格和上下文

目标：让桌宠更像持续存在的角色，而不是一次性问答接口。

任务：

1. 优化 `LLM_SYSTEM_PROMPT` 默认模板。
2. 新增会话存储。
3. 扩展 `ChatRequest` 支持 `session_id`。
4. 扩展 `ChatResponse` 返回 `session_id`。
5. 增加清空会话接口。
6. 补充上下文测试。

产出：

1. 多轮对话更连续。
2. 角色风格更稳定。

### P3：coding intent 统一到 `/runs`

目标：通过 LangGraph 统一 coding intent、runs 链路和角色事件，避免聊天入口、实验 Agent 和 runs 各走各的。

任务：

1. 重构 `agent_workflow` 为最小 LangGraph Agent Brain。
2. 新增 Intent Router Node。
3. 新增 Chat Node，复用现有 LLM client 和 conversation store。
4. 新增 Coding Node，负责把用户请求转成 run 输入。
5. 新增 Run Tool Node，复用现有 run service。
6. 新增 Roleplay Node，复用 `character_interface`。
7. 调整 `chat_action/coding.py`，让 coding intent 进入 LangGraph，而不是直接走独立逻辑。

产出：

1. LongCat 作为 LLM provider 被 LangGraph 节点调用。
2. LangGraph 成为真正的 Agent Brain。
3. 代码任务统一走成熟的 runs 生命周期。
4. 前端可以通过同一套消息机制展示任务进度。

### P4：路由拆分与工程整理

目标：降低入口复杂度，提高后续维护性。

任务：

1. 新增 `backend/app/api/`。
2. 拆分 health、llm、chat、runs、messages routes。
3. 保持 `main.py` 只负责 app 编排。
4. 整理 `mock_backend.py` 的位置或用途。
5. 更新 README。

产出：

1. 后端入口更清晰。
2. 团队成员更容易定位接口实现。

### P5：更细工具能力

目标：为后续真实 Agent 能力打基础。

任务：

1. 增加读取项目文件工具。
2. 增加列目录工具。
3. 增加运行测试工具。
4. 增加错误摘要工具。
5. 所有工具继续受 workspace 安全边界约束。

产出：

1. 后续 Agent 可以更安全地处理真实项目任务。
2. 不需要让 LLM 直接拼危险命令。

## 7. 推荐执行顺序

建议按以下顺序推进：

1. P0：消息协议收口。
2. P1：角色事件服务。
3. P2：聊天人格和上下文。
4. P3：coding intent 接入 `/runs`。
5. P4：路由拆分和 Agent 模块整理。
6. P5：工具能力扩展。

其中，最优先的是 P0 和 P1。

原因是它们最贴合 AI 桌宠项目目的，也最方便前端同学尽快联调。

## 8. 当前阶段的判断

当前后端已经具备可运行、可测试、可继续扩展的基础。

下一步不应该优先追求复杂多 Agent，而应该先完成：

1. 桌宠消息协议。
2. 角色表现策略。
3. 聊天上下文。
4. LangGraph Agent Brain 与 `/runs` 的统一。

这样项目会从“有 AI 接口的前端应用”，推进到“真的像 AI 桌宠的本地应用”。

## 9. 当前执行进度

截至 2026-05-07，已完成：

1. P0：消息协议收口。
   已将 `motion` 纳入正式后端消息协议，并更新 `MessageEnvelope`、`MessageSender`、协议文档和测试。

2. P1：角色事件服务基础版。
   已新增 `character_interface.py` 和 `character_action/`，统一封装聊天和任务运行中的桌宠表现事件。

3. P2：聊天人格和上下文基础版。
   已优化默认 `LLM_SYSTEM_PROMPT`，新增轻量 `conversation_store.py`，并为 `/chat` 增加 `session_id` 输入输出和会话清空接口。

4. P3：LangGraph coding intent 基础版。
   已将 `agent_workflow` 从线性演示流程收敛为最小 LangGraph 图，包含 `router`、`chat_node`、`coding_node`、`run_tool_node`、`unknown_node` 和 `roleplay_node`。
   当前 `/chat` 的 coding intent 会进入 LangGraph，由 `run_tool_node` 复用现有 run service 创建 run，并返回 `run_id`。

5. P3：LangGraph 普通聊天接入版。
   已将普通 `chat intent` 也纳入最小 LangGraph 图，当前 `/chat` 的普通聊天和 coding 都统一经过 `agent_workflow`。
   会话记忆、LLM client、聊天生命周期事件仍保持现有行为，同时避免把同一条聊天正文通过 HTTP 响应和 `agent:chat` 队列重复输出。

6. P3：`/runs` 最终结果消息闭环版。
   已在 run 成功、失败、取消和异常结束时，统一生成简洁的任务总结文本并写入 `agent:chat`。
   当前 coding intent 在 `/chat` 返回 `run_id` 后，前端还可以继续通过 `/messages` 收到最终任务结果，而不会重复收到初始说明文本。

7. P3：LangGraph run 结果总结接入版。
   已新增独立的 `run_summary_graph`，用于读取终态 run record、生成用户可读总结，并统一发送最终 `agent:chat`。
   当前 LangGraph 已不再只负责“创建 run”，也开始承担“消费 run 结果并做结果总结”的职责；若总结节点不可用，则自动回退到确定性摘要文本。

8. P3：LangGraph repair decision 接入版。
   已新增独立的 `repair_decision_graph`，用于在单次执行失败后分析失败摘要、判断是否允许进入自动修复，并向现有 `/runs` repair 流程回传结构化决策。
   当前自动修复已经不再只是 service 层里的裸条件判断，而是先经过 LangGraph 的失败分析和 repair decision，再决定是否继续生成修复脚本。

9. P3：LangGraph repair generation 接入版。
   已将 repair graph 从“只做决策”继续扩展到“决策 + 生成修复脚本”。

## 10. 队长反馈后的架构修正

队长指出：`LongCat` 是大模型本身；项目真正需要的是 `LangGraph`，因为 LangGraph 才能组织 Agent 的状态流和工具调用。

本计划据此修正为：

1. `LongCat` 只作为 LLM provider，不再把“接入 LongCat”等同于“完成 Agent”。
2. `LangGraph` 作为下一阶段核心主线，用来承载 Intent Router、Chat Node、Coding Node、Run Tool Node、Roleplay Node。
3. 现有 `/runs`、safe tools、message protocol、character events、conversation store 都保留，作为 LangGraph 节点可以调用的能力。
4. P3 不再只是“coding intent 直接接 `/runs`”，而是“coding intent 进入 LangGraph，再由 LangGraph 调用 `/runs` 和工具层”。
5. 多 Agent 公司化架构后置，先实现一个最小可用 LangGraph Agent Brain。

修正后的最小闭环：

```text
用户输入
  -> /chat
  -> LangGraph router
  -> chat_node 或 coding_node
  -> run_tool_node 或 llm_node
  -> roleplay_node
  -> /messages + ChatResponse
```
