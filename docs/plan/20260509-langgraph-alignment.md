# 2026-05-09 后端对齐修正计划

## 1. 背景

这份计划用于在 `docs/plan/20260506-2.md` 的基础上，结合最近整理出的材料，重新校准后端下一阶段的优先级。

新的判断依据主要来自 3 个方面：

1. 当前仓库里的真实后端实现。
2. 已经整理过的后端接口与模块文档。
3. 队长新增的方向约束：
   - `quip` 要能更频繁、更细粒度地输出“现在在干什么”
   - LangGraph 节点开头可以直接发送 JSON
   - 图片输入场景要提前考虑
   - token 消耗需要控制
   - 面向用户的输出不应长期停留在工程化 URL / 调试口径

本计划的目标不是推翻旧方案，而是把旧方案里仍然正确的部分保留下来，并把已经偏离桌宠产品目标的部分重新排序。

## 2. 与原方案的融合度和差异度

这里的“原方案”主要指 [`docs/plan/20260506-2.md`](./20260506-2.md)。

### 2.1 融合度高的部分

以下判断与原方案高度一致，应继续保留：

1. `LangGraph` 应作为主线编排层，`LongCat` 只作为 LLM provider。
2. `chat / runs / messages / character events / tools` 这些能力不应各走各的，而应被统一收口到 Agent Runtime。
3. `character_interface`、`messaging`、`message_queue` 这条“角色表现消息链路”是项目主线，不是附属功能。
4. `/runs` 仍然是代码任务执行与恢复的基础能力层，不应被临时逻辑替代。
5. `conversation_store`、`run_store`、`tools/`、`agent_workflow/` 的分层方向仍然成立。
6. 多 Agent 公司化协同不应抢占当前主线，当前仍应坚持“单主图、少节点、可观测”的最小可用路线。

### 2.2 方向一致但需要重排优先级的部分

原方案里有些内容方向没错，但当前优先级需要调整：

1. 原方案强调了内部事件模型、trace、diagnostics，这些能力仍然有价值，但不应继续压过“桌宠实时表现”。
2. 原方案持续推进了 `P8 / P9` 的 diagnostics 和 support 拆分，这对维护性有帮助，但已经不是当前最紧迫的产品差距。
3. 原方案已经提出 Roleplay Node、角色事件服务、消息协议收口，这些方向正确；但当前更关键的是“节点开头就发消息”，而不是只在收口节点发消息。
4. 原方案已经保留了 `motion` 协议和角色动作链路，这一点与队长要求一致；但现在还缺“什么时候发动作”的稳定规则。

### 2.3 差异度较大的部分

以下内容说明原方案和当前产品目标之间存在明显偏差，后续需要修正：

1. 当前很多用户可见输出仍然直接暴露 `GET /runs/{run_id}`、`/snapshot`、`/attempts` 这类 URL。
   这更像开发期调试助手，而不是桌宠对话体验。

2. 当前 `quip` 主要绑定在粗粒度生命周期事件上，例如：
   - `chat.started`
   - `run.started`
   - `run.finished`
   这还不是“开始写代码、开始 debug、开始读取状态”这种节点级反馈。

3. 当前 LangGraph 节点大多只返回状态，不在节点入口直接抛前端可消费的 JSON。
   从桌宠表现角度看，这会让前端只能看到“结果”，看不到“过程”。

4. 当前 token 控制仍然偏弱。
   - 上下文会持续拼接
   - 意图识别仍可能额外触发一次 LLM 调用
   - 缺少按 token 预算进行裁剪或统计的机制

5. 当前聊天 schema 仍是纯文本输入，缺少图片输入的正式结构。
   如果前端后续传截图或图片，当前后端没有稳定承接面。

6. 当前 `detect_intent()` 仍然过于像“前置语义分类器”。
   这会导致很多正常自然语言输入在没有命中既有关键词时，被迫走向 `unknown` 或不断依赖补丁式特判。

### 2.4 应降级或后置的内容

以下内容不能说没价值，但在当前阶段不应继续前置：

1. 继续大幅扩展 diagnostics 复杂度。
2. 继续把更多工程调试信息直接暴露给最终用户。
3. 过早引入复杂多 Agent 协同。
4. 继续围绕通用平台能力做扩张，而不是围绕桌宠主线做收口。

## 3. 重新校准后的核心判断

当前后端的核心问题，不是“没有工作流”，而是“工作流已经有了，但桌宠表现还没有真正嵌入工作流节点”。

因此下一阶段的重点应该从：

- 继续增加 support/helper/diagnostics 细节

转为：

- 把 LangGraph 的节点级状态转成前端可直接消费的桌宠表现消息

一句话概括：

> 现阶段后端最应该补的是“过程中的表现层”，而不是继续堆“结果后的解释层”。

## 4. 新计划的执行原则

### 4.1 产品目标优先于工程调试友好

后端仍要保留 diagnostics，但用户真正感知到的是：

1. 桌宠有没有在思考
2. 桌宠现在是不是在写代码 / 查状态 / 修复问题
3. 桌宠的回复是不是自然，而不是直接甩接口 URL

### 4.2 节点级事件优先于末端收口事件

后续表现消息的重点应从：

- `任务开始了`
- `任务完成了`

逐步前移为：

- `开始理解你的请求`
- `开始读取工作区`
- `开始创建任务`
- `开始查看任务状态`
- `开始尝试修复`

### 4.3 用户可见文案和机器可见状态分离

后端应继续保留结构化状态，但不要把面向调试的 URL、node name、内部接口路径直接当作用户回复正文。

### 4.4 控制 token 消耗是功能正确性的一部分

本项目不是通用聊天网页，而是本地桌宠。响应成本和上下文体积必须尽早收口，否则后续体验会持续劣化。

### 4.5 借鉴统一入口思想，但不照搬 CLI coding agent

可以借鉴 Kimi Code 一类 coding agent 的一个关键优点：

1. 不把所有用户自然语言先硬分成很多语义意图。
2. 而是让统一 agent 主入口决定“直接回答”还是“调用工具”。

但本项目不能照搬它的整体产品形态，因为当前主线仍然必须保留：

1. `LangGraph` 作为 agent 编排层。
2. `/runs` 作为代码任务执行底座。
3. `message_queue / messaging / character_interface` 作为桌宠表现出口。

因此后续应借鉴“统一自然语言入口”的思想，同时把前置 `intent` 层收缩成窄路由，而不是继续膨胀成规则表。

## 5. 新阶段计划

## P1：节点级表现消息接入

目标：

让 LangGraph 在关键节点入口就能发出简短、稳定、结构化的桌宠消息。

任务：

1. 为节点入口新增统一的消息发射 helper。
2. 在以下节点开头补发 JSON 消息：
   - `chat_node`
   - `coding_node`
   - `workspace_tool_node`
   - `run_tool_node`
   - `run_snapshot_node`
   - `run_control_node`
3. 每条节点入口消息至少支持：
   - `quip`
   - `status`
   - 可选 `expression`
   - 可选 `motion`
4. 统一消息字段，避免每个节点手工拼接。

建议落点：

- `backend/app/services/character_interface.py`
- `backend/app/services/character_action/events.py`
- `backend/app/agent_workflow/agent_graph.py`
- `backend/app/agent_workflow/roleplay.py`

完成标准：

1. 前端能看到“正在做什么”的过程提示，而不只是最终结果。
2. 节点入口消息不需要依赖最终 roleplay 收口才出现。

## P1.5：统一自然语言入口与窄路由收口

目标：

减少越来越假的前置意图分类器，把更多正常自然语言输入送入统一 agent 主线。

任务：

1. 将 `detect_intent()` 收缩为窄路由器，只保留稳定、确定性的前置分流。
2. 将正常自然语言输入默认视为可对话请求，而不是优先导向 `unknown`。
3. 让“是否需要实际操作项目/文件/run”成为比“是否像编程话题”更重要的判断标准。
4. 保持 `LangGraph` 负责内部状态分发，不把所有路由负担压在 chat service 的前置分类上。
5. 为简单数学、概念解释、身份/记忆问题、明确操作请求补充回归测试。

建议落点：

- `backend/app/services/chat_action/intent.py`
- `backend/app/services/chat_interface.py`
- `backend/app/agent_workflow/agent_graph.py`
- `backend/tests/test_chat_intent.py`

完成标准：

1. 正常自然语言输入不再频繁落入 `unknown`。
2. 前置分类逻辑不再继续靠补丁式规则膨胀。
3. 项目继续沿着 `LangGraph + /runs + 桌宠表现消息` 主线推进，而不是漂向通用 CLI agent。

## P2：用户态输出去工程化

目标：

把当前偏工程化的对外文案改成桌宠产品口径。

任务：

1. 清理正文里直接暴露的 `GET /runs/...`、`/snapshot`、`/attempts` 等接口式表达。
2. 将“如何继续查看状态”改为更自然的用户说明。
3. 将“工程诊断信息”保留在：
   - diagnostics
   - trace
   - 开发文档
4. 将“用户可见输出”保留为：
   - 当前在做什么
   - 当前结果是什么
   - 下一步建议是什么

建议落点：

- `backend/app/agent_workflow/agent_text_support.py`
- `backend/app/services/run_action/formatters.py`
- `docs/backend-api-specification.md`
- `backend/README.md`

完成标准：

1. 用户对话正文不再像 API 手册。
2. 调试信息仍能通过文档和 diagnostics 获取。

## P3：token 与上下文消耗收口

目标：

减少“问几个问题就烧很多 token”的情况。

任务：

1. 审查 `detect_intent()` 的 LLM 兜底调用频率，尽量让更多常见输入只走规则分流。
2. 为上下文拼接增加更明确的预算策略，而不只是字符截断。
3. 增加简单的用量观察字段，至少能知道：
   - 是否触发了 LLM intent 分类
   - 当前上下文是否命中压缩摘要
   - 当前上下文是否被裁剪
4. 对常见短问句、闲聊、简单控制语句优先采用轻量路由策略。

建议落点：

- `backend/app/services/chat_action/intent.py`
- `backend/app/storage/conversation_store.py`
- `backend/app/llm/client.py`

完成标准：

1. 常见聊天输入不应轻易触发多次模型调用。
2. 后端能更明确解释“为什么这次 token 用得多”。

## P4：图片输入能力预留

目标：

为后续截图 / 图片输入预留正式接口，不急着一次做完整多模态。

任务：

1. 扩展聊天请求 schema，允许后续传入图片附件描述。
2. 先定义稳定结构，例如：
   - `attachments`
   - `type`
   - `url` 或 `data`
   - `mime_type`
3. 在未真正启用多模态模型前，先做好：
   - 参数校验
   - 占位降级
   - 明确错误提示
4. 避免前端未来直接绕过 schema 乱传图片字段。

建议落点：

- `backend/app/schemas.py`
- `backend/app/services/chat_interface.py`
- `backend/app/llm/client.py`
- `docs/backend-api-specification.md`

完成标准：

1. 后端正式知道“图片输入长什么样”。
2. 即使暂未启用图片理解，也不会出现接口层混乱。

## P5：URL / 资源表达收口

目标：

把“接口路径”与“用户可见资源提示”分开。

任务：

1. 重新定义 run 相关结果中的“查看详情”表达。
2. 如果后续还需要给前端传资源定位信息，应优先走结构化字段，而不是写进聊天正文。
3. 对可能继续变化的 URL 形式做统一出口，避免散落在多处文案里。

建议落点：

- `backend/app/agent_workflow/agent_text_support.py`
- `backend/app/services/run_action/formatters.py`
- `backend/app/api/route_support.py`

完成标准：

1. URL 变化不需要到处改文案。
2. 用户态文本与机器态定位字段解耦。

## P6：保留但降级的工程收口项

目标：

把原计划中仍有价值但不应抢主线的部分保留为次优先任务。

任务：

1. 继续维护 diagnostics 的稳定性，但不再优先扩展复杂度。
2. 保留 `P8 / P9` 这类去冗余成果，不再把它们当作最近两轮的主攻方向。
3. 只有在新一轮节点消息和用户态输出稳定后，再考虑继续细化：
   - runtime observability
   - 更细的 trace enrich
   - 更多 helper 拆分

## 6. 推荐执行顺序

建议后续推进顺序：

1. 先做 `P1`：节点级表现消息接入
2. 再做 `P1.5`：统一自然语言入口与窄路由收口
3. 再做 `P2`：用户态输出去工程化
4. 再做 `P3`：token 与上下文收口
5. 然后做 `P4`：图片输入能力预留
6. 最后做 `P5`：URL / 资源表达收口
7. `P6` 作为持续性维护项，穿插处理

## 7. 与旧计划的对应关系

为了避免后续继续混用两套优先级，这里给出简化映射：

1. 新 `P1` 主要承接旧计划中的：
   - 角色事件服务
   - 消息协议收口
   - Roleplay Node

2. 新 `P2` 主要修正旧计划中偏工程化的用户输出问题。

3. 新 `P3` 主要补足旧计划里没有前置处理的 token 成本问题。

4. 新 `P4` 是对旧计划缺失的图片输入场景补位。

5. 旧计划中的 `P8 / P9` 不删除，但从“主线”降级为“维护项”。

## 8. 当前结论

当前后端并没有走错大方向。

真正的问题是：

1. LangGraph 已经接进来了，但节点级表现消息还没真正接进去。
2. 消息链路已经有了，但更多还是服务于生命周期节点，而不是服务于过程体验。
3. 诊断和工程结构已经做了不少，但桌宠产品感最关键的那一层还没有完全补齐。

因此下一阶段不应继续漂向“更通用、更平台化”，而应继续回到：

- LangGraph 节点
- 桌宠表现消息
- 用户态输出
- token 成本
- 图片输入预留

这 5 条主线。
