# 2026-05-12 Agent Architecture Realignment Plan

副标题：智能体架构重对齐方案 / Agent Architecture Realignment Plan

## 1. 本方案的定位

这是一次架构级修订，不是普通功能迭代计划。

原因：

- 队长要求后续后端修改必须严格对齐 `docs/main_guide/` 下的设计文档。
- 我们重新确认了 Claude Code、Codex 一类工具的关键机制：真正的 agentic loop 不应该是一个全局大 loop 承担所有事情，而应该存在于更内层的 worker/subgraph 中。
- 当前后端已经有一个稳定的 `/chat -> Agent Loop` 主线，但如果继续把 PM、Coder、Tool Executor、QA、Debugger、Roleplay 全部塞入这个大 loop，会逐步偏离设计文档。

本方案用于修正 `docs/plan/20260511-5.md` 后续阶段的方向。

结论：

```text
保留现有 Agent Loop 作为 Turn Controller。
停止把它扩成万能大 loop。
将 coding/debug/tool 的真实循环下沉到内部 Coding Workflow Subgraph。
```

## 2. 设计依据

### 2.1 来自主设计文档的约束

`docs/main_guide/AI Agent 架构优化V2 - Google Docs.md` 强调：

- 不能只有一个全局 `messages` 状态。
- State 必须拆分为前端展示状态和后台工程状态。
- Roleplay Agent 的上下文必须与后台代码、报错、stdout/stderr 物理隔离。
- PM Agent 只做需求拆解，输出 `tasks_list` 后休眠。
- Coder Agent 不读取全局报错历史，只读取项目背景和当前 Task。
- Tool Executor 是普通 Python 函数节点，不是 LLM。
- QA Agent 只读 `raw_error`，产出 `error_summary`，并清除 `raw_error`。
- Debugger Agent 是“即用即抛”的局部节点，只读取 `current_code + error_summary`。
- 后续可研究 LangGraph Send API 或 Subgraphs，实现局部状态隔离。

`docs/main_guide/AI Agent 开发框架与蓝图 - Google Docs.md` 强调：

- 前端桌宠是表现层，不应承接后台工程细节。
- 中间通信层必须能传持续状态，不只是 HTTP 单次返回。
- 后端是“一人公司”式多智能体工作流。
- PM / Architect / Coder / QA 有明确 SOP。
- 不要试图让一个 Prompt 解决所有问题。
- 工具调用和本地执行必须经过沙箱或安全边界。

`docs/main_guide/AI桌宠全栈开发工作流程指南 - Google Docs.md` 强调：

- 四层架构：Frontend、Bridge、Agent Brain、Native Environment。
- WebSocket 或消息总线传输强类型 JSON。
- 状态包至少包含状态更新、角色对话、高危授权请求。
- LangGraph 状态应拆分前台展示状态和后台工程状态。
- 高风险终端命令需要 HITL 审批。
- 原生安全防线包括路径约束、命令黑名单、timeout。

### 2.2 对当前后端的判断

当前后端已经完成的正确部分：

- `/chat` 已收敛到单 Agent Loop 主线。
- 旧 route graph 和 `AGENT_RUNTIME_MODE` 已移除。
- `workflow.completed / workflow.failed` 终态事件已经稳定。
- `ActionPlan` 已开始统一顶层 plan 结构。
- workspace 工具、run 工具、消息协议、安全文件系统已有基础。
- 测试已分类，当前后端具备较好的回归保护。

当前后端需要纠正的方向：

- `agent_loop_graph.py` 现在承担了太多未来扩张压力。
- 如果继续在顶层 Agent Loop 里添加 PM、Coder、QA、Debugger，会形成新的大泥球。
- 当前状态字段仍混有对前端、工具、run、workspace、恢复逻辑的多种职责。
- coding/debug 的循环还没有自己的局部状态边界。
- Roleplay 虽然有消息输出层，但还没有从状态结构上保证不会接触后台脏上下文。

## 3. 核心架构决策

### ADR-1：顶层 Agent Loop 改称 Turn Controller

现有 `backend/app/agent_workflow/loop/agent_loop_graph.py` 不应继续被理解为“完整智能体大脑”。

它的新定位是：

```text
Turn Controller
```

职责：

- 接收一轮 `/chat` 输入。
- 建立 turn_id、session_id、runtime metadata。
- 做轻量 intent gate。
- 选择 chat / coding / diagnostic / confirmation 等内部工作流。
- 发出 node/action/terminal 事件。
- 保证每轮请求一定进入 `workflow.completed` 或 `workflow.failed`。

非职责：

- 不直接承载 PM/Coder/QA/Debugger 的完整工作流。
- 不把长篇 raw error 放入全局状态。
- 不把 Roleplay 和工程状态混在同一上下文里。
- 不在顶层无限 replan。

### ADR-2：真正的 coding loop 下沉到 Coding Workflow Subgraph

未来 coding 相关循环应落在：

```text
backend/app/agent_workflow/coding/
```

建议结构：

```text
agent_workflow/
  coding/
    __init__.py
    state.py
    coding_graph.py
    pm_node.py
    coder_node.py
    executor_node.py
    qa_node.py
    debugger_node.py
    output.py
```

内部流程：

```text
PM
-> Coder
-> Tool Executor
-> QA
-> Debugger
-> Tool Executor
-> Finish / Failed / Auth Required
```

说明：

- PM 只拆需求，不参与后续循环。
- Coder 只拿项目背景和当前 Task。
- Tool Executor 不调用 LLM，只执行受控工具。
- QA 摘要错误并清理 raw error。
- Debugger 只拿 `current_code + error_summary`，修完即丢。

### ADR-3：状态必须分区，禁止一个全局 messages 滚到底

目标状态结构：

```text
TurnState
  turn_id
  session_id
  user_input
  intent
  terminal_status

FrontendState
  ui_status
  current_phase
  roleplay_line
  expression
  motion
  progress
  auth_request

ConversationState
  recent_messages
  compressed_summary
  external_context_preview

RuntimeState
  action_plan
  workflow_trace
  node_events
  stop_reason
  max_steps

EngineeringState
  tasks_list
  current_task
  target_files
  current_code_or_patch
  raw_error_ref
  error_summary
  repair_count
  artifact_refs

ToolState
  tool_name
  tool_input
  tool_result
  tool_error_code
  requires_confirmation
```

强约束：

- `FrontendState` 不允许包含 raw stderr、长代码、完整测试日志。
- `ConversationState` 不允许无限增长。
- `EngineeringState.raw_error` 不进入 Roleplay。
- `raw_error` 优先保存为 artifact/ref，QA 读取后压缩为 `error_summary`。
- Debugger 输入只允许 `current_code_or_patch + error_summary + current_task`。

### ADR-4：Roleplay 是前台代理，不是工程 Agent

Roleplay 的输入只允许：

```text
user_intent
ui_status
current_phase
short_summary
auth_request
```

Roleplay 的输出只允许：

```text
quip
chat line
expression
motion
frontend status
```

Roleplay 禁止读取：

```text
raw_error
full stdout/stderr
full code diff
tool internal stack trace
long conversation dump
```

### ADR-5：工具执行和权限审批必须在 Native/Sandbox 边界

已有能力继续保留：

- `safe_fs.py`
- `safe_execute_command.py`
- `workspace_tools.py`
- confirmation gate
- desktop export 配置开关

后续增强方向：

- `Auth_Request` 成为标准事件。
- 高风险 action 可以暂停工作流。
- 用户确认后通过 resume/continuation 机制继续。
- raw command 永远不能绕过 safe executor。

## 4. 新目标架构

### 4.1 物理层次

```text
Frontend / Desktop Pet
  Live2D, chat panel, auth dialog, status display

Bridge / Messaging
  WebSocket or polling-compatible JSON event stream

Backend Turn Controller
  /chat entry, intent gate, workflow dispatch, terminal event

Workflow Subgraphs
  chat workflow
  coding workflow
  diagnostics workflow
  roleplay workflow

Worker Nodes
  PM
  Coder
  Tool Executor
  QA
  Debugger

Native Tool Boundary
  safe fs
  safe command
  workspace
  optional sandbox
```

### 4.2 `/chat` 调用链

目标链路：

```text
/chat
-> Turn Controller
-> intent gate
-> if chat:
     chat workflow
     roleplay output
     terminal
-> if coding:
     coding workflow subgraph
     roleplay/status output
     terminal / auth_required / failed
-> if unknown:
     clarification workflow
     terminal
```

### 4.3 Coding Workflow Subgraph 调用链

目标链路：

```text
coding.start
-> pm_node
   input: user_requirement
   output: tasks_list

-> coder_node
   input: project_context + current_task
   output: code_patch / file_plan

-> executor_node
   input: code_patch / tool_plan
   output: tool_result / raw_error_ref

-> qa_node
   input: raw_error_ref + current_task
   output: error_summary
   side effect: clear raw_error from active state

-> debugger_node
   input: current_code_or_patch + error_summary + current_task
   output: revised_patch

-> executor_node
-> finish / failed / auth_required
```

```text

```

暂时保留。

- run 生命周期、retry、rerun、cancel、attempt、summary 已有大量测试保护。
- 直接推倒风险过高。

```text

```

也就是说：

```text

```

而不是：

```text

```
